# 算法说明与改进记录

## 1. 当前算法定位
当前求解器属于“约束感知的构造式启发算法 + 轻量元启发式搜索”。

它不是精确算法，也不是成熟的大邻域搜索，而是面向当前框架阶段的一套可运行方案：

- 先构造一个合法装箱方案
- 再通过若干邻域动作反复改进方案
- 用目标函数比较新旧方案

当前同时支持两类目标：

- `max totalFillRate`
- `min totalCost`

## 2. 当前算法结构

### 2.1 单容器装箱内核
位置：
- `packingLogic.py`

职责：
- 给定一个容器和一个待放货物，判断某个候选放置是否合法
- 返回结构化反馈，而不是只给最终对错

当前会检查：
- 旋转是否合法
- 是否越界
- 是否重叠
- 是否超载
- 支撑来源是否合法
- 支撑面积是否足够
- 承压是否超限
- 顶部安全间隙是否足够
- 定向件投影约束是否满足

核心接口：
- `evaluatePlacement(...)`
- `findBestPlacement(...)`

### 2.2 方案构造器
位置：
- `optimization.py`

职责：
- 给定一批货物顺序，逐件构造一个 `ShipmentPlan`
- 优先往已有容器里放
- 放不下再新开容器
- 在多种容器规格下，结合目标函数选择更合适的容器类型

核心接口：
- `buildPlanFromOrder(...)`

### 2.3 目标评估器
位置：
- `optimization.py`
- `solutionValidator.py`

职责：
- 统一计算方案指标
- 根据 `objective` 生成排序依据

核心接口：
- `evaluateObjective(...)`
- `buildPlanRankKey(...)`
- `computePlanMetrics(...)`

### 2.4 外层搜索器
位置：
- `optimization.py`

当前已有两类：
- `greedy_search`
- `simulated_annealing`

它们的共同点是：
- 都围绕一个当前方案搜索邻域
- 都用目标函数比较方案优劣

## 3. 当前邻域动作
这是这轮算法增强的重点。

当前外层搜索已经不再只做“货物顺序交换”，而是支持多种邻域：

### 3.1 `swap_order`
交换两个货物在输入顺序中的位置，然后重新构造方案。

特点：
- 最基础
- 实现简单
- 对单箱问题比较直接

### 3.2 `relocate_item`
把一个货物从当前容器移除，再尝试插入另一个容器。

特点：
- 开始直接修改“容器方案”
- 比单纯换顺序更接近成本优化目标

### 3.3 `change_container_type`
选中一个已有容器，把其中货物用另一种容器类型重新装一遍。

特点：
- 直接搜索“箱型选择”
- 对最低成本问题更有针对性

### 3.4 `close_container`
尝试关闭一个低利用率容器，把其中货物重新塞入其他容器。

特点：
- 直接减少容器数量
- 对总成本下降通常最敏感

## 4. 改进历史

### 阶段 1：单容器 baseline
早期版本的思路更接近：
- 给定一个容器
- 给定一组货物
- 按货物顺序尝试装箱
- 用 `volumeRate` 作为核心评价指标

问题：
- 只适合单容器问题
- 不适合多容器成本目标
- 不理解标签约束

### 阶段 2：框架重构
完成了：
- `ProblemInstance`
- `ShipmentPlan`
- 多容器结构
- 目标函数抽象
- 约束感知装箱内核

此时算法已经能处理：
- 多种容器
- 成本目标
- `standard / fragile / oriented`

但外层搜索还主要依赖货物顺序。

### 阶段 3：当前算法增强
本轮已完成：
- 外层搜索引入容器级邻域动作
- 从“只调货物顺序”开始转向“直接调装箱方案”

当前搜索器已经会尝试：
- 换货物顺序
- 跨容器移货
- 换箱型
- 关箱

这说明算法已经从“顺序优化器”开始向“方案优化器”过渡。

## 5. 当前优点

### 5.1 约束处理集中
所有硬约束都尽量集中在单容器装箱内核处理，不需要每个算法单独理解：
- 易碎件
- 定向件
- 承压
- 顶部间隙

### 5.2 目标切换方便
同一套算法框架可以用于：
- 满容率优先
- 成本优先

### 5.3 已具备方案级搜索雏形
当前已经不再是纯粹“换顺序再重装”的单一模式。

## 6. 当前不足

### 6.1 邻域动作还比较少
虽然已经开始支持容器级动作，但还不够丰富。

例如还没有系统支持：
- 双向换货
- 容器合并后的局部重排
- 多步联动动作

### 6.2 局部重建能力还有限
当前部分动作已经是“局部改方案”，但还没有形成系统的增量更新框架。

很多时候仍会重新构造较大范围的方案。

### 6.3 候选位置剪枝还不够强
当前装箱内核已经有基础候选位置机制，但面对大规模 case 时仍然偏慢。

### 6.4 失败原因还没真正用于引导搜索
虽然内核能返回：
- `reasonCode`
- `supportAreaRatio`
- `bearingPressure`
- `topClearanceCm`

但高层搜索还没有系统利用这些信息来调整方向。

## 7. 下一步改进方向
如果继续做算法效率和解质量优化，建议优先顺序如下：

1. 增强容器级邻域动作
2. 加强局部重建，减少全量重算
3. 优化候选位置生成与剪枝
4. 利用失败原因做搜索引导
5. 尝试两阶段求解：先粗分配，再细装载

## 8. 一句话总结
当前算法已经从“单纯优化货物顺序”升级到“开始直接优化装箱方案”，但仍然处在第一轮增强阶段。

如果后续重点转向算法效率，那么核心目标应当是：

- 从“顺序驱动”进一步走向“容器方案驱动”
- 从“全量重建”进一步走向“局部重建”
